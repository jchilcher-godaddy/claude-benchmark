We Tested Every "Best Practice" Against How Developers Actually Prompt (6,480 Benchmarks)

JC
Johnathen Chilcher Senior SRE, TechLoom

We tested every "best practice" against how developers actually prompt. The gap is 1.4 points across 6,480 runs. Most advice isn't worth the tokens.

This is the capstone. Over the past several weeks, we've run six separate experiments—CLAUDE.md profiles, chain-of-thought, model selection, politeness, context pollution, persona assignment—totaling over 10,000 benchmark runs. Each experiment isolated one variable. This experiment stacks them all together and asks the only question that matters: does the evidence-based approach actually beat how developers naturally prompt?

We built eight prompt variants representing the full spectrum: from zero-setup ("just ask Claude") to stacking every piece of common advice we could find. Then we ran each variant across 3 models, 12 tasks, and 30 repetitions. 6,480 total runs. Here's what happened.

The Cast of Characters

Eight variants. Each represents a real developer archetype or a specific combination of advice from our experiments.

Variant Description What It Tests
bare-default Zero setup. Just ask Claude. No system prompt, no prefix, default temperature. The control. How good is Claude out of the box?
terse-coder temp=0. Nothing else. The "pragmatic developer" who read one forum post. The most common single tweak developers make.
prompt-engineer CoT prefix ("Think step by step") + verbose system prompt with style, architecture, error handling, and testing rules. ~500 tokens. The developer who follows widely-repeated advice.
kitchen-sink Stacks ALL common advice: detailed CoT prefix + temp=0 + the same verbose system prompt. What happens when you combine every "best practice" at once?
best-practices Evidence-based: warm framing ("Could you please"), temp=1.0, code-reviewer persona. ~30 tokens of system prompt. No CoT. Our experiment-derived optimal stack.
tuned-haiku best-practices + Haiku-specific structural guidance (type hints, brief plan, single responsibility). Model-specific tuning for Haiku.
tuned-sonnet best-practices + Sonnet-specific minimal guidance ("clean, idiomatic Python"). Model-specific tuning for Sonnet.
tuned-opus best-practices + Opus-specific simplicity guidance ("elegant simplicity, readability over cleverness"). Model-specific tuning for Opus.

The first four variants represent how developers actually prompt in the wild. The last four represent what our data says you should do. Let's see if the data-driven approach earns its keep.

The entire spread is 1.4 points. From the best variant (tuned-haiku, 94.8) to the worst (kitchen-sink, 93.4), the gap is 1.4 points on a 100-point scale. Across 6,480 runs. Every piece of prompting advice you've ever read fights over a 1.4-point window.

Let that sink in. The developer who just fires off a bare prompt (93.9) lands 0.1 points behind the developer who spent hours crafting a verbose system prompt with CoT and style rules (93.9). The "prompt-engineer" variant literally ties with doing nothing.

But the averages don't tell the whole story.

What the Averages Hide

LLM Quality: Where the Real Gap Lives

The composite score blends multiple rubric dimensions. When we isolate LLM Quality—the dimension that measures how well the model understood and solved the problem—the picture shifts.

Variant LLM Quality
tuned-haiku 95.8
tuned-sonnet 94.7
best-practices 94.0
bare-default 92.8
prompt-engineer 92.6
terse-coder 92.5
tuned-opus 91.3
kitchen-sink 90.7
The LLM Quality gap is 5.1 points (tuned-haiku 95.8 vs. kitchen-sink 90.7). The evidence-based variants consistently produce higher-quality solutions. The composite score compresses this because other dimensions (formatting, completeness) are less sensitive to prompting strategy.

The best-practices family (best-practices, tuned-haiku, tuned-sonnet) dominates the top three LLM Quality slots. The "common advice" family (terse-coder, prompt-engineer, kitchen-sink) clusters at the bottom. The signal is there—it's just masked in the composite.

Three things jump out:

Refactoring is where best-practices wins. best-practices scores 92.0 on refactoring vs. bare-default's 89.2 (+2.8 points). tuned-haiku reaches 93.7 (+4.5 over bare). Meanwhile, kitchen-sink collapses to 86.1—a full 7.6 points below tuned-haiku. If you refactor code with AI, how you prompt matters.
Kitchen-sink destroys refactoring performance. At 86.1, it's the worst refactoring score in the table by a wide margin. Stacking verbose instructions + CoT + temp=0 actively confuses the model on tasks requiring structural judgment. tuned-opus owns instruction-following. At 95.9, it's the highest instruction score across all variants. Opus with a simplicity-focused persona excels at precise specification compliance. But it pays for this by lagging on bug-fix (94.4) and refactoring (88.7).

The pattern is consistent with everything we've seen across experiments: bug-fix and code-gen are "solved" task types where prompting barely moves the needle. Refactoring and instruction-following are where strategy matters—and where the wrong strategy can actively hurt.

The Ranked Playbook

Based on 6,480 runs across this experiment and the 10,000+ from preceding experiments, here is every piece of prompting advice ranked by impact. Tier 1 first.

Tier 1: Free Wins (Stop Doing Things That Hurt)

Stop stacking instructions.

kitchen-sink is the worst variant at 93.4 composite and 90.7 LLM Quality. It stacks CoT + temp=0 + verbose system prompt—exactly what forum posts recommend. Each piece of advice is individually marginal. Together they're actively harmful. The verbose system prompt adds ~600 tokens of noise. The CoT prefix fragments reasoning. temp=0 eliminates sampling diversity. Combined, they produce the worst LLM Quality score and the worst refactoring performance (86.1) in the entire experiment.

Evidence: kitchen-sink: composite 93.4, stdev 8.4, LLM Quality 90.7, refactor 86.1. Worst on three of four metrics. 6,480 runs.

Don't force temp=0.

"Set temperature to 0 for deterministic coding" is everywhere. In our data, terse-coder (temp=0, nothing else) scores 94.0—virtually identical to bare-default (93.9). The temperature sweep found temp=1.0 marginally best with the lowest standard deviation. temp=0 doesn't hurt much on its own, but it doesn't help either, and it's one of the ingredients that makes kitchen-sink toxic.

Evidence: terse-coder 94.0 vs. bare-default 93.9 (+0.1). Temperature sweep: temp=1.0 lowest stdev. 6,480 + temperature sweep runs.

Skip chain-of-thought for code.

prompt-engineer includes "Think step by step" and scores 93.9—identical to bare-default. Our dedicated CoT experiment found every CoT variant scored lower than bare across all models. Opus suffered the most (-1.14 for detailed CoT). CoT works for math puzzles. It fragments the holistic pattern recognition that code generation requires.

Evidence: prompt-engineer 93.9 = bare-default 93.9. CoT experiment: every variant negative, Opus -1.14. 6,480 + 270 CoT runs.

Tier 2: Small Edge (Do These Things)

Use a code-reviewer persona.

"You are a meticulous code reviewer" is the single system prompt instruction in our best-practices variant. The persona sweep experiment found code-reviewer adds +1.0 composite and +2.9 on refactoring. It's the only persona that consistently helped across all models and task types. It primes the model toward correctness and edge-case coverage without constraining its approach.

Evidence: Persona sweep: code-reviewer +1.0 composite, +2.9 refactoring. best-practices (which uses code-reviewer) LLM Quality 94.0 vs. bare-default 92.8 (+1.2). Persona sweep + 6,480 runs.

Use warm framing.

"Could you please" as a prompt prefix. The politeness sweep found warm framing adds +1.5 points and produces the best floor scores across models. It's not about being polite to the AI—it's about framing the task as a collaborative request rather than a terse command. The framing cue shifts the model's response distribution toward more careful, considered output.

Evidence: Politeness sweep: warm framing +1.5 composite, best min scores. Politeness sweep runs.

Use default temperature (1.0).

Leave temperature at 1.0. Don't set it to 0. Our temperature sweep found 1.0 marginally best with the lowest standard deviation. The sampling diversity at temp=1.0 means the model explores more solution space, occasionally finding better approaches. At temp=0, it always takes the highest-probability path—which is usually good but never surprising.

Evidence: Temperature sweep: 1.0 lowest stdev, marginally highest composite. Temperature sweep runs.

Tier 3: Conditional (Model/Task Specific)

Tune Haiku with structural guidance for refactoring.

tuned-haiku adds type hints, brief planning, and single-responsibility reminders to the best-practices base. It barely beats best-practices overall (+0.1 composite), but it reaches 93.7 on refactoring—a +4.5 point lift over bare-default's 89.2. Haiku benefits from structural scaffolding because its internal model of code architecture is less developed. If you use Haiku for refactoring (and you should—it's cheaper), this guidance is worth the tokens.

Evidence: tuned-haiku: composite 94.8, refactor 93.7. bare-default refactor: 89.2. Delta: +4.5 on refactoring. 360 runs.

Use tuned-opus for instruction-heavy tasks.

tuned-opus scores 95.9 on instruction tasks—the highest instruction score across all variants. Opus with "prioritize correctness and elegant simplicity" excels at precise specification compliance. But it's mediocre elsewhere (94.4 bug-fix, 88.7 refactor). Use it when the task is about following a spec, not when it's about restructuring code.

Evidence: tuned-opus instruction: 95.9 (best). tuned-opus refactor: 88.7 (below average). 360 runs.

Target refactoring specifically.

Refactoring shows the biggest variance across approaches: 86.1 (kitchen-sink) to 93.7 (tuned-haiku), a 7.6-point spread. Bug-fix ranges only 4.3 points. Code-gen only 2.6. If you're going to invest effort in prompt optimization, invest it on refactoring tasks. That's where the ROI lives.

Evidence: Refactor spread: 7.6 points. Bug-fix spread: 4.3. Code-gen spread: 2.6. Instruction spread: 7.1. 6,480 runs.

Tier 4: Not Worth It

Verbose style/architecture rules.

prompt-engineer packs ~500+ tokens of style rules, architecture principles, error handling guidelines, and testing advice into the system prompt. Result: 93.9—identical to bare-default. All that effort, all those tokens, zero gain. This is consistent with our CLAUDE.md experiments where we found r=-0.95 between instruction token count and quality. Claude already knows how to code. Telling it again doesn't help.

Evidence: prompt-engineer 93.9 = bare-default 93.9. CLAUDE.md experiment: r=-0.95 token-to-quality correlation. 6,480 + 1,458 runs.

temp=0 "for deterministic coding."

No benefit. terse-coder (temp=0 only) scores 0.1 above bare-default—within noise. And when combined with other advice (kitchen-sink), it contributes to the worst overall result.

Evidence: terse-coder 94.0 vs. bare-default 93.9 (+0.1, within noise). 6,480 runs.

CoT scaffolding for code generation.

"Think step by step" does nothing for code. "Before writing any code, analyze the requirements step by step" does nothing for code. Save the tokens.

Evidence: prompt-engineer (CoT prefix) = bare-default. kitchen-sink (detailed CoT) = worst variant. 6,480 runs.

tuned-haiku has the tightest variance (stdev 4.7) by a wide margin. It's not just the best average—it's the most consistent. kitchen-sink has the widest variance (8.4) and the worst average. More instructions don't just fail to help; they make output less predictable.

For automated pipelines, tuned-haiku is the standout. Highest average, lowest variance, cheapest model. If your use case is batch code generation where consistency matters more than peak performance, tuned-haiku on Haiku is the clear winner.

The Cost Story

Every token in a system prompt costs money and context window. Here's what you're paying for each approach.

Variant Extra Tokens Composite Delta LLM Quality Delta Cost per Point
bare-default 0 — — —
best-practices ~30 +0.2 +1.2 25 tokens/point
prompt-engineer ~500+ +0.0 -0.2 Infinite (no gain)
kitchen-sink ~600+ -0.5 -2.1 Negative (pays to hurt)
kitchen-sink costs ~600 tokens to make output worse. You're paying for a 0.5-point composite drop and a 2.1-point LLM Quality drop. That's not "not worth it"—it's actively destructive. Every token in that verbose system prompt is working against you.

best-practices costs ~30 tokens—a single sentence of system prompt and a three-word prefix. For that, you get +0.2 composite and +1.2 LLM Quality. That's the most efficient prompt investment in the entire dataset. The ratio of signal to token cost is unmatched.

Model-Specific Results

How does best-practices perform relative to bare-default on each model?

Model bare-default best-practices Delta
Haiku 94.7 95.3 +0.6
Sonnet 93.1 94.2 +1.1
Opus 93.8 92.9 -0.9
Opus resists the best-practices template. It's the only model where best-practices scores below bare-default (-0.9). This is consistent with our model selection findings—Opus has strong internal preferences and pushes back against prompting that doesn't align with them. Sonnet benefits the most (+1.1), confirming that even models we previously described as "best left alone" respond to the right minimal intervention.

The model-tuned variants exist precisely because one-size-fits-all doesn't work. tuned-opus adjusts the persona toward simplicity rather than meticulous review, which aligns better with Opus's internal preferences. The lesson: if you use multiple models, your prompt strategy should vary with them.

Cross-Experiment Synthesis

This is the capstone. Here's how every experiment we've run connects to these results.

Experiment Key Finding How It Shows Up Here
CLAUDE.md Profiles r=-0.95 between instruction tokens and quality. Empty beats every profile. prompt-engineer (500+ tokens of instructions) ties bare-default. kitchen-sink (600+ tokens) is worst.
Chain of Thought Every CoT variant hurt. Opus suffered most (-1.14). prompt-engineer's "Think step by step" adds nothing. kitchen-sink's detailed CoT is part of the toxic cocktail.
Model Selection Models have distinct response profiles. One-size-fits-all is suboptimal. Opus resists best-practices (-0.9). tuned-haiku leads on consistency. tuned-opus leads on instruction tasks. Each model needs different guidance.
Politeness Sweep Warm framing ("Could you please") adds +1.5 points with best floor scores. best-practices uses warm framing as its prefix. It's part of why best-practices family outperforms on LLM Quality.
Context Pollution Irrelevant context degrades refactoring by up to -7.5 points. kitchen-sink's verbose system prompt acts as context pollution. Its refactoring score (86.1) is the worst in the table, consistent with the -7.5 finding.
Persona Sweep Code-reviewer persona: +1.0 composite, +2.9 refactoring. Only persona that helped across all models. best-practices uses code-reviewer as its sole persona. It's the backbone of the evidence-based approach.

Every experiment pointed in the same direction: less is more, specificity beats verbosity, and the few things that help are small, targeted interventions. This capstone confirms that the individual findings compose correctly—the evidence-based stack works as expected when assembled, and the common-advice stack fails as predicted.

Limitations

  • Single-file tasks. All 12 tasks are self-contained. Real-world coding involves navigating multi-file codebases where project-specific CLAUDE.md context (build commands, project structure) adds value we can't measure here.
  • Generic coding tasks. These are domain-agnostic problems. Tasks with specialized domain requirements, unfamiliar APIs, or complex business logic may respond differently.
  • Single-turn only. Each run is one prompt, one response. In multi-turn conversations, system prompt tokens compound—making the cost story even more favorable for minimalist approaches.
  • Claude models only. All runs used Claude Haiku 4.5, Sonnet 4.6, and Opus 4.6. GPT, Gemini, and open-source models may respond very differently to these same strategies.
  • LLM-as-judge scoring. Code quality is assessed by another LLM. While our rubric is calibrated and consistent, it introduces a layer of subjectivity that human review would not.
  • 30 reps per cell. With 30 repetitions per variant-model-task combination, we have reasonable statistical power for detecting medium effects, but small effects (under ~0.5 points) may be noise.

What a Good Prompt Actually Looks Like

After 6,480 runs and 8 experiments, here's the concrete setup that performed best. Copy this and modify it for your project. The entire "optimization" is 3 lines of configuration and a 1-sentence persona.

The System Prompt (CLAUDE.md)

# Project: inventory-api

- Stack: Python 3.12, FastAPI, SQLAlchemy 2.0 (async), PostgreSQL 16
- Build: `make dev` / `make test` / `make lint`
- Source: src/inventory/ | Tests: tests/ (mirrors source tree)
- Auth: Bearer tokens validated by middleware in src/inventory/auth.py
- IDs: ULIDs (python-ulid), not UUIDs
- All timestamps UTC, stored as timestamptz, serialized ISO 8601
- API contract: {"data": ..., "meta": {}, "errors": []}
- Currently migrating from sync to async endpoints (tracked in MIGRATION.md)

You are a meticulous code reviewer. Focus on correctness, edge cases, and maintainability in every line you write.

That's it. No style rules, no "use snake_case," no "think step by step," no architecture principles. Just project facts Claude can't know from training, plus the one persona that moved the needle.

The Prompt Pattern

Could you please [your actual request here]

Warm framing. Default temperature (1.0). No CoT prefix. The "Could you please" costs 4 tokens and contributed to the best floor scores in the politeness sweep.

Copy-Paste Test Scenario

Want to see this in action? Here's a self-contained test you can run right now. Save this as test_prompt.py and send it to Claude with the system prompt above:

# test_prompt.py — a refactoring task (the category with the biggest variance)
# Ask Claude to refactor this with the system prompt above.

def get_inventory_report(db, warehouse_id, start_date, end_date, include_zero_stock):
    """Generate inventory report for a warehouse within a date range."""
    results = db.execute(
        "SELECT * FROM inventory WHERE warehouse_id = :wid", {"wid": warehouse_id}
    )
    items = []
    for row in results:
        if row.last_updated >= start_date and row.last_updated <= end_date:
            if include_zero_stock or row.quantity > 0:
                category = db.execute(
                    "SELECT name FROM categories WHERE id = :cid",
                    {"cid": row.category_id}
                ).first()
                supplier = db.execute(
                    "SELECT name, lead_time_days FROM suppliers WHERE id = :sid",
                    {"sid": row.supplier_id}
                ).first()
                item = {
                    "sku": row.sku,
                    "name": row.name,
                    "quantity": row.quantity,
                    "category": category.name if category else "Unknown",
                    "supplier_name": supplier.name if supplier else "Unknown",
                    "lead_time": supplier.lead_time_days if supplier else None,
                    "value": row.quantity * row.unit_price,
                    "low_stock": True if row.quantity < row.reorder_point else False,
                    "last_updated": row.last_updated.isoformat(),
                }
                if row.quantity == 0:
                    item["days_out_of_stock"] = (end_date - row.last_updated).days
                items.append(item)

    total_value = 0
    low_stock_count = 0
    out_of_stock_count = 0
    for item in items:
        total_value = total_value + item["value"]
        if item["low_stock"]:
            low_stock_count = low_stock_count + 1
        if item["quantity"] == 0:
            out_of_stock_count = out_of_stock_count + 1

    return {
        "warehouse_id": warehouse_id,
        "period": {"start": start_date.isoformat(), "end": end_date.isoformat()},
        "items": sorted(items, key=lambda x: x["value"], reverse=True),
        "summary": {
            "total_items": len(items),
            "total_value": total_value,
            "low_stock_count": low_stock_count,
            "out_of_stock_count": out_of_stock_count,
        }
    }

Send this prompt:

Could you please refactor get_inventory_report in test_prompt.py? It has N+1 query
problems, deep nesting, and mixes data access with business logic. Keep the return
contract identical.

What to Look For

Compare the output against a bare prompt (no system message, no "Could you please"). In our benchmarks, the differences showed up most clearly in:

  • LLM quality — best-practices scored 94.0 vs. bare-default's 92.8. The code reads more like a human wrote it.
  • Refactoring specifically — 92.0 vs. 89.2 (+2.8 points). The evidence-based setup produces cleaner decomposition.
  • Consistency — run it 5 times each way. The best-practices output varies less. Fewer "what was it thinking?" outliers.
The entire optimization is ~40 tokens of system prompt beyond project context. One persona sentence, one 4-token prefix, and the discipline to leave everything else out. That's what 6,480 benchmarks say is worth doing.
